## Problem statement

ContactsView expand ▸ và FriendsView hiện đọc cùng `friends` table nhưng:
- 2 query shape khác nhau (`AGGREGATE_INCLUDE` vs `/friends-db` include) → drift risk
- 5 sync paths không thống nhất → user phải click "↻ Đồng bộ Zalo" mới refresh
- `friend_event` listener có duplicate dead code, không emit cho FE refresh
- Alias / display name / avatar đổi từ Zalo native app KHÔNG có event → CRM stale tới khi reconnect
- "Tất cả nick" mode trong FriendsView trả empty (TODO)

Goal: child row của ContactsView CHÍNH LÀ Friend của FriendsView, dữ liệu live-sync từ SDK qua cron + event + on-connect, không phụ thuộc nút manual.

## Architecture decisions (D1-D11)

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| D1 | Sync approach | **B Incremental Unified** | Cover edge cases mà không phá kiến trúc |
| D2 | Sync module location | `backend/src/shared/.../friend-sync-service.ts` mới | DRY, testable, symmetric với alias-sync.ts |
| D3 | Cron strategy | 1 cron `*/15 * * * *` sequential per-account + 200ms stagger | Pattern giống zalo-health-check, ổn định CPU |
| D4 | Canonical serializer | `backend/src/shared/friend-serializer.ts` export `FRIEND_INCLUDE` + `toFriendDto` | Single source of truth, type-safe |
| D5 | Socket event | `friend:updated` với `{friendId, contactId, zaloAccountId, patch}` | Instant cell update, no refetch |
| D6 | All-nicks mode | Flat per-pair list (`/friends-db/all-nicks`) | Đúng bản chất Friend = per-pair |
| D7 | Manual button | Đổi label → "↻ Làm mới ngay" + cooldown 5s | Escape hatch giữ control cho user |
| D8 | Sync error logging | `logActivity({systemSource:'friend_sync_error'})` trong catch | Observable, dashboard-ready |
| D9 | Test scope | Backend Vitest 100% + FE manual + TODO setup FE Vitest | Tránh balloon, BE đủ catch regression |
| D10 | Cron emit overhead | Diff-then-emit (chỉ emit khi field actually changed) | 99% rows no change → 0 emit |
| D11 | TODOs deferred | Per-field syncedAt + adaptive cron | Optimization sau khi monitoring |

## Implementation order (parallelization)

| Lane | Steps | Modules | Notes |
|------|-------|---------|-------|
| **A (foundation)** | friend-serializer.ts → AGGREGATE_INCLUDE reuse | `shared/`, `modules/contacts/` | Block all others |
| **B (sync service)** | friend-sync-service.ts → diff-then-emit logic | `modules/zalo/` | Depends on A |
| **C (event cleanup)** | Remove dup listener, add emit to friend_event handler | `modules/zalo/zalo-listener-factory.ts` | Independent |
| **D (cron)** | friend-sync-cron.ts + wire in app.ts + zalo-pool autoSyncOnConnect | `modules/zalo/`, `app.ts` | Depends on B |
| **E (routes)** | /friends-db FRIEND_INCLUDE + cooldown + /all-nicks | `modules/zalo/friend-routes.ts` | Depends on A |
| **F (FE socket)** | use-friend-socket.ts | `frontend/composables/` | Independent of BE if mock |
| **G (FE wiring)** | FriendsView + ContactsView subscribe + UI updates | `frontend/views/` | Depends on F |

**Execution**: Launch C + F in parallel với A. Sau khi A xong → B → D, E. Sau F xong → G. Total 3 lanes, ~2-3 ngày sequential / ~1.5 ngày parallel.

**Conflict flag**: Lane B và Lane C cùng touch `zalo-listener-factory.ts` (B emit patch in handleFriendEvent, C remove dup listener) → cẩn thận merge.

## TODOs added to TODOS.md

### TODO: Per-field syncedAt + adaptive cron (Approach C-lite)
**What**: Add `Friend.aliasSyncedAt`, `avatarSyncedAt`, `nameSyncedAt` columns. Cron query stale fields selective thay vì full pull every 15 min.

**Why**: After monitoring shows fixed 15min cron either too aggressive (wasted API calls) or too lazy (stale data complaints). Per-field tracking enables surgical refresh.

**Pros**:
- Lower Zalo SDK call volume (only pull stale fields)
- Lower socket emit volume
- Better observability (per-field freshness)

**Cons**:
- Schema migration (3 columns + indexes)
- Sync logic complexity (per-field branching)
- Needs monitoring infra first (else premature)

**Context**: This is approach C-lite minus outbox pattern. Plan B (current) uses fixed 15min cron + full pull. If monitoring shows >50% cron cycles cause 0 emits → consider this. Triggered by: cron metric `friend_sync_cycles_zero_diff_pct > 50%` for 7 days.

**Depends on**: Monitoring dashboard for cron emit stats (also future TODO).
